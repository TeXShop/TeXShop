Sincronización código-pdf (pdfsync)

La sincronización es una importante utilidad incorporada en Textures, una implementación comercial de TeX para Macintosh Classic desarrollada por Blue Sky TeX Systems. Al usar la sincronización, un autor puede hacer clic en la ventana del dvi previsto e, inmediatamente, el cursor aparece en el correspondiente lugar de la ventana de código. Blue Sky ha prometido una versión nativa para MacOS X de Textures, pero todavía no la ha lanzado.

La implementación de la sincronización en Textures es admirada por su habilidad para localizar el correspondiente punto en el código. He oído que Blue Sky TeX Systems reescribió TeX para proporcionar la información necesaria para esta utilidad, pero no conozco detalles de su implementación.

Desde entonces, varios sistemas TeX en otras plataformas han incorporado, aproximadamente, la misma utilidad. Para ello han usado estilos adicionales que también hacían que TeX escribiera información extra en el archivo dvi. En verano de 2003, Piero D'Ancona escribió un archivo de estilo similar en cierto sentido, pero para pdflatex. Este paquete, pdfsync.sty, hace que pdftex escriba, al componer páginas, un archivo extra, en el cual guarda la información necesaria para lograr la correlación entre los archivos de entrada (código tex) y salida (pdf). El trabajo de D'Ancona depende de comandos internos de pdftex que sacan las coordenadas x e y de ciertas operaciones de composición.

Recientemente, Jérôme Laurens se unió al proyecto de D'Ancona, contribuyendo con cambios al paquete pdfsync. Entonces, Laurens implementó la sincronización con el pdf en iTeXMac; su implementación es la primera y —actualmente— la más robusta que funciona en un interfaz de TeX para Mac OS X.

TeXShop 1.34 comienza el proceso de traer estos desarrollos a TeXShop. Para usar pdfsync, hay que buscar el archivo pdfsync.sty y colocarlo en ~/Library/texmf/tex/latex. Con el tiempo, este archivo será parte de la distribución de TeX de Gerben Wierda. La sincronización que proporciona pdfsync sólo esta disponible para pdflatex y context; no funciona si componemos por medio del motor opcional tex + ghostscript.

Para usar pdfsync, añade al preambulo de tu documento con código tex (antes de la línea con el \begin{document}) la siguiente línea:

\usepackage{pdfsync}
Este paquete será ignorado si estás compilando con tex + ghostscript, pero si lo estas haciendo con pdflatex, se creará un archivo extra con extansión .pdfsync. Por ejemplo, si el documento con el códido fuente se llama main.tex, el archivo extra se llamará main.pdfsync. Este archivo contiene la información necesaria para relacionar puntos del archivo pdf con puntos del archivo tex. Cuando se usa pdfsync, la mayoría de los archivos se compilan sin errores, pero, en un pequeño número de casos, este paquete introduce errores extra. Por favor, comunica tales casos a D'Ancona, Laurens o a mí.

Una vez que has compuesto las páginas a partir de un archivo con pdfsync, haz clic en un punto del pdf previsto mientras mantienes pulsada la tecla comando. Aparecer‡ en el frente la ventana con el código, allí estará seleccionada la línea que contiene el correspondiente código, y la ventana se desplazará hasta mostrar esa línea si fuera necesario. Si un proyecto contiene un archivo raíz y capítulos adicionales añadidos con \include, y si el punto en la ventana del pdf corresponde al código incluido en el archivo de algún capítulo, el archivo se abrirá, aparecerá en el frente, y se producirá el desplazamiento hasta el lugar apropiado en el código.

En esta implementación inicial, la línea de código que se muestra será aproximadamente la correcta, pero la real puede encontrarse una pocas líneas más arriba o más abajo. El archivo pdfsync de D'Ancona contiene información adicional para localizar el punto de manera más precisa; TeXShop incorporará el código necesario para usar esta información tan pronto como sea posible.

Actuamente, pdfsync no funciona si el código tex está distribuido en varios archivo por medio de comandos \input. Es decir, \include funciona, pero \input no. Es posible que esta restricción sea eliminada en el futuro.
